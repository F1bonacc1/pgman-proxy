## Fixture state

Verifications T130–T136 ran against a 3-peer process-compose rig where
node-b was in `failed` state (2/3 healthy). This is a stricter fixture
than the spec's "healthy 3-peer cluster":

- The aggregator fan-out has one unreachable sibling, adding to the
  worst case for `pgmctl status` latency.
- The dump's per-peer slice for node-b is recorded as `failed`, so
  the dump emitter exercises its partial-outcome path.
- The doctor's `nodes.no-failed-state` and `postgres.responding`
  checks fire correctly (TPR fixture).

A full all-healthy baseline (for an additional FPR measurement)
requires `process-compose process restart node-b`, which the auto-
mode classifier blocks pending explicit operator authorization.
